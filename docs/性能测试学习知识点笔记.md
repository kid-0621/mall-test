# 性能测试学习知识点笔记

---

## 一、什么是性能测试？

> 性能测试不是"能不能跑通"，而是"跑得快不快、扛不扛得住"。

| 维度 | 功能测试（你已会） | 性能测试 |
|------|------------------|---------|
| 问什么 | 接口能不能正常返回？ | 1000 个人同时访问会不会崩？ |
| 看什么 | HTTP 状态码、业务 code | 响应时间、吞吐量、CPU、内存 |
| 测多久 | 每条用例 0.1 秒 | 持续加压 5-30 分钟 |
| 工具 | requests + pytest | JMeter / Locust |

## 二、核心指标（面试必问）

### 2.1 三大黄金指标

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   响应时间（RT / Response Time）                      │
│   └── 用户发起请求 → 收到完整响应的时间                │
│   └── 目标：P95 < 2 秒（95% 的请求在 2 秒内完成）     │
│                                                     │
│   吞吐量（TPS / QPS）                                │
│   └── 每秒处理多少个请求                              │
│   └── TPS = 并发数 ÷ 平均响应时间                     │
│   └── 比如：100 并发、平均 0.2s → TPS = 500          │
│                                                     │
│   错误率（Error Rate）                               │
│   └── 失败请求 ÷ 总请求 × 100%                       │
│   └── 目标：< 0.1%（生产环境）                        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.2 其次看什么

| 指标 | 说明 | 哪里看 |
|------|------|--------|
| **并发用户数** | 同一时刻有多少人在操作 | 压测工具设置 |
| **P50/P90/P95/P99** | 50%/90%/95%/99% 的请求在多少毫秒内完成 | 压测报告 |
| **CPU 使用率** | 服务器 CPU 打满没 | `top` / `htop` / Grafana |
| **内存使用率** | 内存有没有泄漏 | `free -m` / Grafana |
| **磁盘 IO** | 读写是否成为瓶颈 | `iostat` / Grafana |
| **网络带宽** | 流量有没有打满网卡 | `iftop` / `nload` |

### 2.3 响应时间分位数通俗理解

```
100 次请求的响应时间（毫秒）：
200, 210, 220, 230, ... 480, 490, 5000  ← 有一个慢请求！

P50 = 350ms  → 一半的请求在 350ms 以内
P95 = 490ms  → 95% 的请求在 490ms 以内（好的！）
P99 = 4800ms → 99% 的请求在 4.8 秒内（那 1% 拖后腿）

结论：P95 好用不代表没问题，P99 暴露了长尾慢请求！
```

## 三、性能测试类型

```
负载测试（Load Test）
    └── 模拟预期正常流量，比如 500 并发跑 10 分钟
    └── 目的：确认系统在日常负载下 OK

压力测试（Stress Test）
    └── 不断加压直到系统崩溃
    └── 目的：找到系统的极限吞吐量和拐点

稳定性测试（Endurance / Soak Test）
    └── 中等负载持续跑 8-24 小时
    └── 目的：发现内存泄漏、连接池耗尽等长期问题

峰值测试（Spike Test）
    └── 瞬间流量暴增（比如秒杀）
    └── 目的：验证系统能否应对突发流量

基准测试（Benchmark）
    └── 1 个用户访问 100 次
    └── 目的：拿到单用户下的基线数据，对比用
```

## 四、性能测试流程（5 步法）

```
步骤 1：需求分析
    ├── 搞清楚线上预期有多少用户？
    ├── 哪些接口是高频核心接口？
    └── SLA 是什么？（比如 P95 < 500ms）

步骤 2：场景设计
    ├── 登录占比 10%，浏览商品 60%，下单 20%，支付 10%
    ├── 思考时间（Think Time）要不要加？
    └── 数据准备（1000 个测试账号、商品、订单）

步骤 3：脚本编写
    ├── 录制 or 手写 HTTP 请求脚本
    ├── 参数化（不同用户不同数据）
    ├── 关联（token 提取 + 回传）
    └── 断言（响应时间 + 响应内容）

步骤 4：执行压测
    ├── 先 1 并发基准测试 → 再阶梯加压
    ├── 监控服务器：CPU、内存、DB 慢查询
    └── 找到拐点：TPS 不再增长甚至下降 = 瓶颈

步骤 5：分析报告
    ├── TPS vs 并发数曲线图
    ├── 响应时间随并发变化趋势
    ├── 错误集中在哪里？
    └── 瓶颈在应用层还是数据库？
```

## 五、JMeter vs Locust 对比

### 5.1 本质区别

| 维度 | JMeter | Locust |
|------|--------|--------|
| **开发语言** | Java（GUI 操作） | Python（纯代码编写） |
| **脚本形式** | XML（.jmx 文件） | Python 代码（.py 文件） |
| **并发模型** | 线程池（1 线程 = 1 虚拟用户） | **协程（gevent）**，1 个进程可以模拟数千用户 |
| **资源消耗** | 高（每个线程占用内存） | 低（协程极轻量） |
| **学习方式** | 图形界面拖拽 + 填表 | 写 Python 代码 |
| **分布式** | 支持（Master-Slave） | 支持（Master-Worker） |
| **报告** | GUI 查看 + HTML Dashboard | 自带的 Web UI 实时看 |
| **插件生态** | 极丰富（几百个插件） | 较少，但够用 |
| **录制功能** | ✅ 内置 HTTP 代理录制 | ❌ 无（需手动编写） |

### 5.2 直观对比：同样一个登录接口的脚本

**JMeter（图形界面操作）：**
```
右键 Test Plan → Add → Thread Group
  → 右键 Thread Group → Add → Sampler → HTTP Request
    → 填入 Server: localhost, Port: 8080, Path: /admin/login
    → 填入 Body Data: {"username":"admin","password":"macro123"}
  → 右键 HTTP Request → Add → Listener → View Results Tree
  → 点绿色三角运行
```

**Locust（Python 代码）：**
```python
from locust import HttpUser, task, between

class MallUser(HttpUser):
    wait_time = between(1, 3)       # 每个用户操作间隔 1-3 秒
    token = None

    def on_start(self):
        """每个虚拟用户启动时先登录"""
        resp = self.client.post("/admin/login", json={
            "username": "admin",
            "password": "macro123"
        })
        self.token = resp.json()["data"]["token"]

    @task(3)                        # 权重 3，执行频率更高
    def browse_products(self):
        self.client.get("/product/list", params={"pageNum": 1, "pageSize": 10},
                        headers={"Authorization": self.token})

    @task(1)                        # 权重 1，偶尔下单
    def create_order(self):
        self.client.post("/order/create", json={...},
                         headers={"Authorization": self.token})
```

```bash
# 启动
locust -f mall_performance.py --host=http://localhost:8080

# 打开浏览器 http://localhost:8089 设置并发数，点 Start
```

**一眼看出区别：** JMeter 靠图形界面拖拽，Locust 靠写代码。

### 5.3 关键差异：并发模型

```
JMeter 线程模型：
  100 个线程 → 占用 100 个线程资源
  1000 个线程 → 内存炸了，机器扛不住

Locust 协程模型（gevent）：
  1 个进程 → 可以跑 1000 个虚拟用户
  单机轻松模拟 5000-10000 并发
```

### 5.4 选型建议

| 你的情况 | 推荐 | 理由 |
|---------|------|------|
| 你是 Python 测试工程师 | **Locust** | Python 生态无缝衔接，和 pytest 一脉相承 |
| 团队有专职性能测试 | JMeter | GUI 操作门槛低，任何人都能上手 |
| 想面试加分 | **Locust** | 协程并发模型是面试亮点，JMeter 太普遍了 |
| 要录制浏览器操作 | JMeter | 自带录制器，Locust 没有 |
| 复杂场景（多条业务线混合） | **Locust** | 代码写逻辑灵活度远高于 GUI 拖拽 |
| 给非技术人员用 | JMeter | 图形界面，不需要会写代码 |

## 六、我的推荐：先学 Locust

### 六个理由

**1. 你已经有 Python 基础**
```python
# 你现在的测试代码
def test_login(self, base_api):
    res = base_api.post("/admin/login", data)
    assert res["code"] == 200

# Locust 压测代码——几乎一模一样！
@task
def login(self):
    res = self.client.post("/admin/login", json=data)
    assert res.json()["code"] == 200
```

**2. 协程并发模型是技术亮点**
面试时说"我用 Locust 的 gevent 协程模型，单进程模拟 5000 并发"，比"我用 JMeter 开 5000 个线程"强 10 倍。

**3. 代码即文档**
JMeter 的 .jmx 文件是 XML，Git 没法 diff；Locust 是 Python 文件，Git 能清晰看到每次改动。

**4. CI/CD 天然集成**
```bash
# Locust 可以无头模式运行（不需要 GUI），直接放进 CI
locust -f test.py --headless -u 100 -r 10 -t 5m --csv=report
#  ← 100 用户，每秒加 10 个，跑 5 分钟，输出 CSV 报告
```

**5. 学习成本低**
JMeter：学 GUI 操作 + 学 XML 配置 + 学 BeanShell/JSR223 脚本 + 学插件体系
Locust：学 Python（你已经会了）+ 学几个装饰器

**6. 实际项目更好用**
你当前 mall 项目的 108 条接口用例是用 Python + requests 写的，转 Locust 压测脚本几乎是复制粘贴改几行。JMeter 要从头录一遍。

## 七、Locust 核心知识点速览

### 7.1 安装
```bash
pip install locust
```

### 7.2 核心概念

```python
from locust import HttpUser, task, between

class MyUser(HttpUser):
    # 每个虚拟用户的架构

    wait_time = between(1, 3)
    # ↑ 操作间隔 = 模拟真实用户不会一秒点 100 次

    def on_start(self):
        # 每个虚拟用户启动时执行一次（登录、初始化数据）
        pass

    @task            # 不带权重，默认权重 1
    def action_a(self):
        pass

    @task(5)         # 权重 5，相当于 5 个 action_a 的执行频率
    def action_b(self):
        pass
```

### 7.3 运行方式

| 模式 | 命令 | 用途 |
|------|------|------|
| **Web UI** | `locust -f test.py` → 打开 `http://localhost:8089` | 开发调试、手动调并发 |
| **Headless** | `locust -f test.py --headless -u 100 -r 10 -t 5m` | CI 集成、自动化压测 |

### 7.4 关键参数

```
-u 100     → 总并发用户数 100
-r 10      → 每秒启动 10 个用户（爬坡速率）
-t 5m      → 跑 5 分钟后自动停止
--csv=report  → 输出 report_stats.csv
--html=report.html → 输出可视化报告
```

### 7.5 mall 项目示例：完整压测脚本

```python
# mall_locust.py
from locust import HttpUser, task, between
import random

class MallApiUser(HttpUser):
    wait_time = between(1, 2)

    def on_start(self):
        """登录获取 token"""
        resp = self.client.post("/admin/login", json={
            "username": "admin",
            "password": "macro123"
        })
        data = resp.json()["data"]
        self.token = data["tokenHead"] + data["token"]
        self.headers = {"Authorization": self.token}

    # ── 浏览商品（高频操作，权重高）──
    @task(10)
    def list_products(self):
        self.client.get("/product/list",
            params={"pageNum": 1, "pageSize": 10},
            headers=self.headers)

    # ── 查看订单（中频操作）──
    @task(5)
    def list_orders(self):
        self.client.get("/order/list",
            params={"pageNum": 1, "pageSize": 5},
            headers=self.headers)

    # ── 搜索品牌（中低频）──
    @task(3)
    def list_brands(self):
        self.client.get("/brand/list",
            params={"pageNum": 1, "pageSize": 5},
            headers=self.headers)

    # ── 创建订单（低频操作）──
    @task(1)
    def create_order(self):
        self.client.post("/order/generateOrder", json={
            "orderItemList": [{
                "productId": random.randint(1, 50),
                "productQuantity": 1
            }]
        }, headers=self.headers)
```

## 八、JMeter 核心知识点速览（了解即可）

### 8.1 核心组件

```
Test Plan（测试计划）
├── Thread Group（线程组）
│   ├── 线程数 = 并发用户数
│   ├── Ramp-Up = 爬坡时间（秒）
│   └── 循环次数 = 1 / 永远
│
├── Sampler（取样器）
│   └── HTTP Request：填 URL、Method、Body
│
├── Config Element（配置元件）
│   ├── HTTP Header Manager：统一加 header
│   └── CSV Data Set Config：从 CSV 读测试数据
│
├── Timer（定时器）
│   └── Constant Timer：固定间隔
│   └── Gaussian Random Timer：随机间隔
│
├── Assertion（断言）
│   ├── Response Assertion：检查响应内容
│   └── Duration Assertion：响应时间超过 N ms 就失败
│
├── Listener（监听器）
│   ├── View Results Tree：看每次请求详情
│   ├── Aggregate Report：汇总表（用这个！）
│   └── Summary Report：简要汇总
│
└── Post Processor（后置处理器）
    ├── Regular Expression Extractor：正则提取（用这个做关联）
    └── JSON Extractor：JSON 路径提取
```

### 8.2 命令行运行（CI 用）

```bash
# 非 GUI 模式运行
jmeter -n -t mall_test.jmx -l result.jtl -e -o report/
#       ↑           ↑               ↑          ↑
#    no-gui    脚本文件        结果文件    HTML 报告
```

## 十、面试高频问题

### Q1: "你怎么做性能测试的？"

> 我用 Locust 写 Python 压测脚本，按线上实际流量比例分配各接口的 @task 权重。先在单用户下做基准测试拿到基线 TPS，再阶梯加压（10→50→100→500→1000 并发），观察 TPS 和响应时间的变化趋势。同时用 top、iostat 监控服务器 CPU 和内存，找到系统拐点。最后输出报告，标记瓶颈在哪里。

### Q2: "TPS 上不去了怎么办？"

> 我按照从外到内的顺序排查：
> 1. 先看带宽有没有打满（iftop/nload）
> 2. 再看应用层：连接池够不够、线程池配置、是否有锁等待
> 3. 最后看数据库：慢查询日志、索引是否缺失、连接数是否耗尽
> 通常会加一层缓存（Redis）解决大部分读瓶颈。

### Q3: "JMeter 和 Locust 有什么区别？"

> 最大的区别是并发模型：JMeter 用线程池，一个线程一个虚拟用户，资源消耗大；Locust 用 Python 协程（gevent），单进程能模拟数千并发。另外 JMeter 是 GUI 操作，Locust 是纯代码。我选 Locust 是因为我们的测试框架是 Python 生态，Locust 可以直接复用已有的接口调用逻辑。

### Q4: "你怎么保证压测数据真实？"

> 三方面：第一，用线上真实流量日志分析各接口的调用占比，按比例分配 @task 权重；第二，加上 wait_time 随机间隔模拟用户思考时间；第三，数据用真实量级（比如 1000 个测试用户、10000 条商品），避免缓存命中率失真。
